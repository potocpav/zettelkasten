# What is Functional Programming

Functional programming (FP) is, similarly to other programming paradigms, quite difficult to pin down. According to Wikipedia, FP has the following characteristics:

* Functions are treated as first-class citizens. They are frequently passed as arguments to other functions.
* Larger functions are composed from small functions in a modular manner.
* Functions are deterministic and side-effect-free, similarly to mathematical functions.
* Function definitions are trees of expressions that each return a value, rather than a sequence of imperative statements which change the state of the program.
* Immutable values are preferred over mutable values.

Seeing the value of these characteristics is not easy, however. Each of them must be argued separately, and it can be difficult to make a solid case for their utility, especially to a person who is used to programming in a different paradigm. How can we convince fellow developers that all the functional weirdness is well warranted?

I think there is a deeper truth hidden beneath the usual functional programming characteristics. The rules above can be seen merely as effects of a single underlying rule, and the value of the rule is self-evident. Without further ado, here it is:

<!--
> Functional programming follows from encoding program semantics as precisely as possible using types.
-->
<p style="font-weight: bold; font-size: larger; padding: 0.5rem; margin: 1rem; border: 1px solid black; text-align: center;">
Functional programming is the consequence of using types to precisely encode program semantics.
</p>

If you agree that type systems should be used to their full potential, functional programming is not much of a paradigm - it is rather just a natural consequence. And it is quite uncontroversial to see that type systems should be wielded efficiently to prevent bugs and maximize correctness.

Note that the above statement is an implication:

$$\text{precise types} ⇒ \text{functional programming}.$$

The converse isn't true, as FP is also possible in weakly-typed languages:

$$\text{functional programming} ⇏ \text{precise types}.$$

In the following paragraphs, we will take a look at various characteristics of functional programming and show how they follow from precise typing.

## Functions

First. let's look at functions, in some languages known as subroutines, methods, or procedures. Typed languages mostly specify function interaction with outside world via argument types and return type. Argument types specify the prerequisites, and the return type specifies the effects of the function.

```c++
int add(int a, int b) {
    return a + b;
}
```

This simple model is, however, very restrictive. Sometimes, we want to create a function with prerequisites or effects that are not representable by argument types. To name a few examples:

* Accessing global state, instance state, or `static` variables in C.
* Dependencies on external state, such as the presence of a file or network connection.
* Multiple return values in C.
* External interactions: printing to a console, writing to a database, etc.
* Randomness. Semantics may depend on thermal noise in your processor.
* Exceptions.

Capturing all of these interactions in the type system will make the type system more complicated, and writing code more tedious. Imagine having a Java checked-exceptions-like system, but for every side effect and every argument mutation. But if left unchecked, the bullet points above are holes in the type system, interactions which the compiler knows little about. We must rely on our reasoning that their ordering, multiplicity, and semantics are correct.

## Objects and methods

Objects are collections of data together with methods which operate on said data. The object data, also called state, is either directly accessible in method bodies, or is explicitely passed as an argument.

Functional programming is about **being honest with your types**. Encoding your target domain as precisely as possible using types. Using the type system to its full potential. These are the effect of this decision:

For functions this implies several properties, commonly known as *referential transparency*.

- Functions only depend on their arguments.
- Functions only effect their return values, no side-effects allowed.

If we didn't have referential transparency, there would be interactions not encoded in types, not obvious syntactically and semantically. With referential transparency, functions' interaction with the rest of the program is fully captured by the type system. Functions can be freely composed without compromising their correctness. They can be analysed and tested in isolation.

## Where to stop?

Designing software functionally is extra effort compared to imperative programming. Religiously following the FP paradigm may have diminishing returns, which would mean that there is a point where the extra effort is no longer justified by the benefits. Is there such a point?

The extra effort manifests in three ways:

1. Learing curve. It is hard to learn functional paradigm's more exotic patterns. This is the cost a developer pays only once in their life. Also known as "becoming a better programmer".
2. Planning. Modelling domains precisely with types requires some upfront planning. You may have to sit down with a pen and paper and try to understand your task as well as possible before you start writing code.
3. Day-to-day programming overhead. This is probably negligible.

Perhaps unintuitively, pinning your domain down with precise types doesn't decrease flexibility. Rather, types allow you to encode the requirements of your functions, but nothing extra. Functional code can be amazingly expressive.
